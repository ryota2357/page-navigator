# ブラウザ拡張機能「page-navigator」設定画面のUIデザイン依頼

## 1. プロダクト概要

`page-navigator`(仮名)は、ブラウザ拡張機能で、ユーザーが定義したキーバインドによってWebページ上の操作(スクロール、要素フォーカス、タブ操作等)を発火させるツールです。Vimium や web-search-navigator が近い既存プロダクトです。修飾キーは必須ではなく、`j` のような単独キーでも動作させられます。

設定画面は、拡張機能ポップアップ(アイコンクリック時)ではなく、`chrome-extension://...` 形式の**専用設定ページ**で提供します。ポップアップ側は最小化(現在の有効/無効状態と「設定を開く」ボタンのみ)するため、本依頼の対象は専用設定ページです。

## 2. ターゲットユーザー

Vimium、web-search-navigator など類似拡張の既存ユーザー、およびキーボード操作を好む技術寄りのユーザー。完全な初心者向けの誘導は不要です(技術名そのままUIに出す前提)。ただし各アクションには説明文を表示する動線が必要です。

## 3. 設定の概念モデル(これが最重要)

このUIが表現するべき本質は、**1つのバインディング = (トリガーキー群) × (アクション) × (オプション)** という関係です。

- **Trigger Keys (トリガー)**: 1つのバインディングは複数のキー(組み合わせ)を持てる。例: `j` と `ArrowDown` の両方で同じ動作。タグ形式の入力 (`[j ✕] [ArrowDown ✕] [+]`) を想定
- **Action (アクション)**: 拡張機能が提供する操作。`scrollPage`、`nextTab`、`focusNextResult` のような技術名でユーザーに見せる。アクションごとに説明文を持つ。総数は最終的に100超
- **Options (オプション)**: 各アクションが持つ設定値。型は数値・boolean・選択肢のいずれかに限定される。例: `scrollPage` の `amount: 100`, `inertia: true`
- 同じアクションでもオプションが異なれば**別のバインディング**として扱われる。例: `j → scrollPage(100px)` と `J → scrollPage(500px)` は2つの別ルール

このモデルを破綻なくUIに落とすことが核心です。

## 4. スコープ(適用範囲)

- **Global**: どのページでも有効な汎用バインディング (50+ アクション)
- **Site-specific**: 特定サービス(Google検索結果、GitHub、YouTube 等)のページのみで有効なバインディング
- 動作時の優先: そのページのSite-specificにマッチするバインディングがあればそれを使う。なければGlobalにフォールバック
- Site-specificの対象サービスは**拡張機能側が定義**する(ユーザーが任意ドメインを追加するわけではない)
- 開発初期: 5サービス程度。将来: 20サービス程度

## 5. 全体レイアウト

**左サイドバー + 右メイン領域**の構成を想定しています。サイドバーは「設定項目の種類」ではなく「適用スコープ」で切り替える役割です。

サイドバー構成案:

- 上部: **GLOBAL**
- 中部: **SITE SPECIFIC** — 対応サービスがフラットに並ぶ(カテゴリ階層なし)。サービス名そのまま (Google, GitHub, YouTube...)
- 下部: **SYSTEM** — インポート/エクスポート、ヘルプ、基本設定

「Google と Yahoo で同じ設定にしたい」というユースケースに対しては、**継承の概念は出さず、明示的なコピー操作**で対応する方針です。例えば「Google設定を右クリック→Yahooに複製」のような。具体的なインタラクション設計はデザイナーに委ねます。

Site-specificを「対応サービス全部を常時表示」するか、「ユーザーが何か設定済みのものだけ表示 + サービス追加ボタンを置く」かは決まっていません。デザイン上の判断を求めます。

## 6. バインディング設定UI(右メイン領域)

選択スコープのバインディング一覧をリストまたはカードで表示し、各行が1つの完全なルールを表します。

各ルールの構成要素:

1. **Trigger Keys**: タグ形式の複数入力欄
2. **Action**: オートコンプリート式のピッカー(100+のアクションから選ぶため、絞り込み検索が必須)。アクション名は技術名、説明文も併記
3. **Options**: アクション選択後に動的展開される。型は数値・boolean・選択肢のいずれか

「+ 新規追加」ボタンと、各行の削除アクションが必要です。

## 7. アクションピッカーの要件

- 100以上のアクションから選ぶため、絞り込み検索が必須(部分一致または fuzzy 相当)
- アクション名は技術名で表示、説明文を視覚的に近接配置
- アクションごとの詳細説明を読むための動線(外部ドキュメントへのリンク等)
- **サイト固有アクション**(特定サイトでのみ動作するもの)には視覚的な識別(バッジ・アイコン等)が必要
  - 補足: Globalスコープでは汎用アクションのみ選択可能、Site-specificスコープでは汎用 + そのサイト固有のアクションが選択可能
- ピッカーの形態(ドロップダウン内検索 / モーダル / 別パネル / コマンドパレット)はデザイナーの判断

## 8. 競合警告

同一スコープ内で同じトリガーキーが複数のバインディングに存在する場合:

- リスト上で警告を可視化(赤バッジ等、形態は委ねる)
- ユーザーは編集途中で競合状態のまま操作できる(編集を阻害しない)
- **保存ボタン**を押した時、競合があれば確認ダイアログを表示:
  - メッセージ: 「以下のキーが競合しています。これらはいずれも実行されません。」(競合キーの一覧表示)
  - アクション: 「戻る」「このまま保存」の2択
  - どちらを残すかを選ばせる UI は不要(ユーザーは戻って自分で直す)

## 9. 空状態(Empty State)

デフォルトで全スコープが未設定です。初回および何もない状態のスコープでは、「プリセット読み込み」を主たる導線として案内します。

- プリセット例: 「vim-like」「web-search-navigator-like」等(拡張機能側が用意するJSONを内部で読み込む)
- ワンクリックで適用できる
- このUIはユーザーの**第一印象を決める部分**なので、丁寧に設計してほしい

## 10. インポート/エクスポート

形式は JSON 等のテキストファイル。

**インポートUI:**

- ファイル選択
- ファイル内に含まれるスコープを表示 (例: GLOBAL / Google / GitHub / YouTube)
- 各スコープに対して以下の制御:
  - 「読み込む / 読み込まない」のトグル
  - 「追加(merge)」または「上書き(replace)」の選択
- 適用ボタンで実行

「全部または何もエクスポートしないか」では不便なため、上記の選択UIが必須です。

**エクスポートUI:**

シンプルに「現在の全設定をエクスポート」で十分(受け取った側がインポート時に選択できるため)。ただし「特定スコープのみエクスポート」UIを追加するかはデザイナーの判断。

## 11. その他の機能要件

**サイト固有ページに「このサイトで使えるキーマップ一覧」**:

そのサービスのページに対して有効なバインディングをまとめて確認できる、読み取り専用のリファレンスビュー。GlobalとSite-specificの両方を含む。検索可能だと望ましい。設定編集とは別の場所/ビューとして配置。

## 12. 視覚的方向性

- **ノイズの少ないシンプルさ**を最優先
- 情報量・概念は多いプロダクトだが、それをすっきり見せること自体がデザインの価値
- 参考イメージ: Linear、Raycast、Vercel dashboard、Stripe dashboard のような**高密度ミニマル**(Apple系の余白を贅沢に取るシンプルさとは違う)
- ダークモード対応は不要(将来対応の余地は残してOK)
- フォント、カラーパレット、コンポーネントスタイルはデザイナー提案ベース

## 13. 設計を委ねる箇所(デザイナーの判断を求めます)

以下は決め切れていない、または設計に強く依存するため、判断・提案を求めます:

- Site-specific サイドバーで「対応サービス全部を表示するか、設定済みのみ表示するか」
- 「設定を別サイトに複製」のインタラクション(右クリック / ボタン / ドラッグ など)
- アクションピッカーの形態(インライン / ドロップダウン / モーダル / コマンドパレット)
- 競合警告のビジュアル(バッジの形・色・配置、メッセージの表現)
- 空状態でのプリセット提示の演出
- インポート時のスコープ選択UIの具体形
- 「このサイトで使えるキーマップ一覧」リファレンスの配置(タブ? モーダル? 同一画面の領域?)

## 14. 根本的な代替提案について

依頼者は意図的にリストUI + サイドバーという既存パターンに収束させましたが、もっと適した別のUIパターン(例: キーボード可視化UI、コマンドパレット中心のUI、GUI + テキスト編集ハイブリッド等)があれば、デザインを始める前に提示してください。検討した上で現方針を採用するか、別案に切り替えるか判断します。

ただし、第3節の概念モデル(トリガー × アクション × オプション)と第4節のスコープモデルは、内部仕様として確定しているため変更不可です。

## 15. デザイン依頼の内容

以下を期待します:

- 主要画面のレイアウト(Global設定、Site-specific設定、空状態、インポートUI、リファレンスビュー等)
- 主要コンポーネントの設計(バインディング行、アクションピッカー、トリガー入力、競合警告等)
- 主要なインタラクションの記述(またはプロトタイプ)
- カラーパレット、タイポグラフィ、間隔の体系
- デザインを進める中で出てきた疑問点・提案点の言語化

依頼者はデザインの専門知識がほぼないため、デザイン上の判断にあたっては「なぜそうしたか」の理由を簡潔に併記いただけると助かります。
